# PAD EDIT — コード名候補（Candidates）の考え方

PAD EDIT 画面の **Candidates** 行に表示するコード名は、プリセット内の他パッド名の一覧ではなく、**選択中の Root** と **Input Notes** から推論したラベル候補である。

実装: `JPad/Models/ChordCandidateRecognizer.swift`  
呼び出し: `PadEditorViewModel.candidates`（編集中は `editingNotes`、それ以外は `notes`）

---

## 前提

| 入力 | 意味 |
|------|------|
| **Root** | ルートキーボードで選んだ音名（`Eb` / `Ab` などフラット表記に正規化） |
| **Input Notes** | パッドの `bassNotes` + `chordNotes` をマージした MIDI ノート（エディタで編集可能） |

候補のルートは **常に UI で選んだ Root** に固定する。Input Notes にルート音が含まれていなくても、その Root を `0` 度としてコードタイプを照合する。

---

## 2 種類の推論を併記する

同じ Root に対し、次の **2 通り** でマッチした名前を出し、**1 列にマージ** して表示する。

### 1. 全ボイシング（優先・先に表示）

Input Notes に含まれる **すべてのピッチクラス**（音名・オクターブは無視し 12 音に畳む）から、選んだ Root 基準のインターバル集合を作り、コードテンプレートと照合する。

- ベースのルートオクターブも上声部のテンションも、すべて判定に入る。

### 2. ルートを除いたボイシング（後に表示）

**Root と同じピッチクラス** のノートを Input Notes から除いた残りだけで、同様にインターバル集合を作り照合する。

- ジャズの **ルートレス・コンプ**（ベースにルート、上声部にルート無し）を意識した解釈。
- 上から N 音だけ、といった制限は **かけない**。ルート以外の音種は **すべて** 使う（9th がトップでも、下にある 3rd / 7th を拾うため）。

### 表示順と重複

1. 全ボイシングでヒットした名前（スコア降順）
2. ルート除きでヒットした名前（スコア降順）
3. **同じ文字列は先勝ちで 1 つだけ**（全ボイシング側を優先）

ルート除き側の音種が **1 つ以下** のときは、そのグループからは候補を出さない（コード判定に情報が足りないため）。

---

## コードタイプの照合

固定の **テンプレート一覧**（`maj9`, `m9`, `13`, `7`, `m7b5`, `dim7` など）について、Root からの半音インターバル集合が `required` を満たすかを見る。

- `13` は `required` に加え、`optional`（9th / 3rd / 13th など）が **1 つ以上** 含まれる必要がある。
- 複数テンプレートが一致するときは **priority** と、含まれる extension の数でスコア化し、高い順に並べる。

### ガイドトーンの加点（ジャズ向け）

次の組み合わせが揃っている候補に **+10** 点（3rd + 7th がそろっている解釈を少し優先）。

| 3rd | 7th | 想定 |
|-----|-----|------|
| 短 3 度 (3) | 短 7 度 (10) | マイナー系 |
| 長 3 度 (4) | 長 7 度 (11) | メジャー 7th 系 |
| 長 3 度 (4) | 短 7 度 (10) | ドミナント系 |

---

## プリセットとの関係

`jazz-progression-starter-bass-arp` などは典型的に:

- **MIDI 60 未満** → ベース（ルートのオクターブ）
- **MIDI 60 以上** → コード域（しばしばルートレス）

Pad 保存時（`PadEditorViewModel.save`）も同じ 60 で bass / chord に分割する。候補推論は **保存後のマージノート全体** を見るが、「ルート除き」グループは **音域では切らず**、ルート音種だけを除く。

---

## ユーザー操作との連動

- **Root 変更** → 候補はすべて新しい Root 名で再計算（`ChordLabel.replacingRoot` でラベルも更新）。
- **候補タップ** → `label` をその文字列にし、文字列からパースできれば `root` も更新。
- **Input Notes 編集**（ポップアップ中を含む）→ ノートが変わるたびに候補を再計算。

---

## 既知の限界（甘いところ）

意図的な簡略化と、今後直せる余地を明示しておく。

### 1. 「ルート除く」がコード域限定ではない

ルートと同じ音種を **全音域から** 除くだけである。ベース域にルート以外の音があるプリセットでは、その音も「ルート除き」判定に入る。  
厳密な「60 以上だけを見る」ルートレス判定には **していない**。

### 2. テンション過多で候補が並びやすい

`13` や `9` など、条件の緩いテンプレートが複数同時にマッチすることがある。ガイドトーン加点で多少絞るが、**唯一の正解** を保証するものではない。

### 3. Root 選択ミスに弱い

Input Notes にルートが無くても、**選んだ Root を正** として照合する。Root が実際のベースと違うと、候補名全体がずれる（Notes は合っていても表記が違う、という状態になりうる）。

### 4. 転回形・スラッシュ表記は出さない

例: 上声部が G から始まる Am9 を `Am9/G` のようには表示しない。常に **選んだ Root + サフィックス** 形式。

### 5. プリセットの `role` フィールドは未使用

JSON の `role`（`Imaj9` など）は候補生成に使っていない。和音のピッチクラスのみから判定する。

### 6. 候補の出自を UI では区別しない

全ボイシング由来とルート除き由来は **同じチップ列** に並べ、重複はまとめるだけ。どちらから来たかは画面上では分からない。
